# Implementation Plan: MbDFEM Pipeline in FreeCAD

## Goal Description
The objective is to upgrade our static structural prototype into a functioning **Multibody Dynamics + Finite Element Method (MbDFEM)** simulation pipeline. This requires creating a framework that calculates large-scale kinematic motion over time (MbD) and repeatedly solves for localized internal structural deformation (FEM) caused by inertial forces.

Because FreeCAD 1.0 does not possess a native, tightly-coupled MbDFEM solver, we must construct a custom **Explicit Weak-Coupling Orchestrator** in Python to bridge Assembly/Kinematic data with the CalculiX math engine.

### User Review Required
> [!IMPORTANT]
> **Coupling Architecture Decision:** Do we start with **One-Way Coupling** or jump straight to **Two-Way Coupling**?
> - *One-Way (Recommended for V1):* The MbD engine calculates the swing trajectory. The FEM engine calculates the bending at each frame using the MbD forces. The bending does *not* feed back to alter the swing. 
> - *Two-Way:* The FEM bending physically alters the center of mass, which actively changes the MbD swing arc. This is highly realistic but prone to mathematical instability in early prototypes.

## Proposed System Architecture

To keep the system highly modular, we will build a **generalized** object-oriented framework. This means it will be capable of handling arbitrary CAD models, not just the pendulum!

---

### Phase 1: The Kinematic Engine (MbD)
Instead of relying on unstable CAD physics engines, we will implement a generic Python solver to calculate rigid position arrays.

#### [NEW] `mbd_solver.py`
- **Purpose:** Mathematical engine to solve non-linear multi-body equations.
- **Initial Scope:** Begins with a generalized 1-Degree-of-Freedom (DOF) dynamic solver that takes in arbitrary mass/inertia properties. (The pendulum will be our mathematical "Hello World" to prove it works before throwing complex vehicles or cranes at it).
- **Outputs:** Generic arrays of time steps ($t$), angular positions ($\theta$), velocities ($\omega$), and accelerations ($\alpha$).

### Phase 2: The Generalized FEM Translator
This module takes raw kinematic numbers and translates them into physical loads on *any* specified 3D model.

#### [NEW] `fem_translator.py`
- **Purpose:** FreeCAD scripting module offering a generalized Object API.
- **API Call Example:** `update_body(target_cad_object, position_vector, accel_vector)`
  1. Applies a mathematical transformation matrix to rotate *any* target 3D geometry in the document.
  2. Calculates the dynamic forces ($F_c$, $F_t$) based on the passed vectors.
  3. Translates these forces into local `ConstraintForce` objects on the target mesh.

### Phase 3: The Orchestrator (The Loop)

#### [NEW] `mbdfem_orchestrator.py`
- **Purpose:** The master loop that binds the system together asynchronously.
- **Workflow:** 
  1. Calls `mbd_solver.py` to get the trajectory array.
  2. `for step in trajectory:`
  3. Calls `fem_translator.py` to update the geometries mathematically.
  4. Triggers `ObjectsFem.makeSolverCalculix` to silently write `.inp` files and solve iteratively in the background.

## Open Questions

> [!WARNING]
> FreeCAD GUI stability during loops is notoriously poor. Running a 100-step loop might crash the active UI. 
> **Question:** Should the orchestrator run purely headlessly (in the background, just generating `.frd` files to disk without updating the 3D view), or do you want the UI to visually refresh frame-by-frame as it computes?

## Verification Plan

### Automated Tests
- Create `test_mbd.py` to verify the mathematical kinematics output a perfect trajectory without energy loss.
- Run a 3-frame mini-loop via the orchestrator to confirm CalculiX successfully generates distinct sequential `.frd` files.

### Manual Verification
- We will load the final sequence of `.frd` files into FreeCAD to ensure the dynamic deformation tracks structurally accurate across a simulated timeline.
